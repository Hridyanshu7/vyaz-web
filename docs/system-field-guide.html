01

The one-breath picture

Vyaz lets a reader “talk to a book.” An AI voice narrates a chapter word-for-word and answers questions out loud, in a live back-and-forth — like an audiobook you can interrupt and quiz.

Under the hood there are only a handful of players, and most of the magic is one live phone-call-style connection between the reader's browser and Google's voice AI. Everything else exists to set up that call, feed it the right chapter, keep it alive, and draw what's happening on screen.

Plain version

Think of it as a phone call. The browser is the reader's handset (mic + speaker + screen). Gemini is the narrator on the other end. A small server (an “edge function”) is the switchboard operator who dials the call and hands over the script — but never lets the reader see the phone number (the secret key). The database is the library where the books and settings are kept.

02

The cast of systems

Five things do all the work. Here's each one in a sentence, then we'll watch them cooperate.

The browser

React app on the reader's device

Captures the microphone, plays the narrator's audio, and paints the transcript, waveform and progress bar. Built with React + Vite, hosted on Vercel. This is where all the live audio plumbing lives.

Gemini (Google)

The voice AI brain

Google's Gemini Live model. One connection to it does listening (speech→text), thinking, and speaking (text→voice) all at once. It's the narrator.

Edge functions

Small trusted servers (Supabase)

Tiny programs that run on the server, not the browser. They hold the secret keys and assemble the narrator's instructions. The browser asks them for a “session,” they hand back what's needed.

The database

Supabase / Postgres

Stores books (and their full text), users, sessions, and all the settings (which voice, which model, the prompt). Also handles login and permissions.

Cartesia

Alternate voice provider

A different voice-AI company, wired in as a switchable backup. Not used day-to-day (Gemini is primary), but the plumbing exists. More in §10.

03

One Talk session, start to finish

This is the heart of it. Follow a single reader tapping Talk on a chapter. Each step names the system doing the work.

1

Reader taps “Talk” — browser asks for a session

The browser calls the voice-session edge function, sending the book title, chapter, and the chapter's pre-split sections (the text).

browser → edge function
2

Edge function builds the brief & hands back the keys

It reads settings from the database, stitches the system prompt together (rules + the whole chapter text glued in), and returns the API key, model name, voice name, and prompt to the browser.

edge function ↔ database
3

Browser opens the live connection to Gemini

It opens a WebSocket (a permanent two-way line) straight to Google and sends a “setup” message: the model, voice, the prompt, and all the audio settings (the knobs in §6).

browser → Gemini (WebSocket)
4

Microphone starts streaming

The browser grabs the mic, chops the audio into tiny chunks (PCM at 16 kHz), and streams them up the line continuously.

browser mic → Gemini
5

Gemini narrates — audio + a transcript come back

Google sends back two separate streams: the voice audio (played through the speaker) and a text transcript of what it's saying. They arrive on slightly different timelines.

Gemini → browser
6

Browser paints the screen

Audio is queued and played. The transcript is split into book words (black) vs the agent's own remarks (grey) using the ((double-parens)) markers, and a progress bar advances by matching the spoken words against the chapter.

browser (GeminiLiveModal)
7

Reader speaks — the narrator pauses and answers

Google's voice-activity detection notices the reader talking, interrupts the narration, transcribes the question, answers it, then can resume reading.

full-duplex, both directions at once
8

The line ages out — and silently reconnects

Google caps each connection at ~10–15 minutes. Just before it drops, we grab a resume handle, reopen the line, and continue where we left off — the reader sees a brief “Reconnecting…” countdown, not a dead session.

browser (session resumption)
The one thing to remember

The audio and its transcript are two separate deliveries from Google, not one. Most of our past bugs (missing words, mismatched bubbles) came from treating them as if they arrived together. They don't.

04

Why voice is different from a chatbot

A text chatbot is like sending letters: you write, send, wait, get a reply, repeat. Each message is a fresh, complete request.

Voice here is like an open phone line (“full-duplex” — both people can talk at once). That difference explains almost every quirk:

  • It streams. Words arrive a piece at a time, so the screen updates mid-sentence — and half-finished sentences need careful handling.
  • It can be interrupted. The reader can cut in any moment; the system has to detect that and stop talking. That's what voice-activity detection is for.
  • It has no memory of its own once the line drops. If the connection dies, the “conversation so far” is gone unless we deliberately resume it (§5).
  • Timing races are everywhere. Audio vs transcript, interrupt vs turn-end — things arrive slightly out of order, which is why the code is fussy about when it resets things.
Rule of thumb

If a problem is about what the model says or how it behaves (paraphrasing, tone, forgetting to mark its asides) — that's the prompt's job (§7). If it's about words getting lost, cut, or mis-drawn — that's the browser code's job. Different fixes, different files.

05

Memory & the two hard limits

“Memory” for a voice AI means: how much can it hold in mind at once, and what happens when the line breaks.

What the narrator “knows”

Right now we paste the entire current chapter into the system prompt at the start. So the narrator can read and answer about that chapter — but nothing from other chapters (that would need RAG, currently parked; see glossary). This is called “stuffing the context.”

The context window

The model can only hold so much text at once (its “context window”). A long chapter plus a long back-and-forth can crowd it. We turned on sliding-window compression, which quietly summarizes older turns so the session can run longer without overflowing.

Resumption (surviving a dropped line)

Google periodically hands us a resume handle — like a coat-check ticket for the conversation. If the line drops, we reopen and present the ticket, and Google restores the context. That's how a 30-minute chapter survives 15-minute connections.

Hard limit 1 — connection length

Each WebSocket lasts ~10–15 minutes. Handled now via auto-resumption, so it no longer kills a session — but it's why reconnection logic exists at all.

Hard limit 2 — how many people at once

On Google's simple “Developer API,” only ~3 live sessions can run per key at the same time. That's the real wall for mass usage — the 4th simultaneous listener gets rejected. The fix is moving to Vertex AI (1,000 at once), which is a bigger infrastructure change and not done yet. Note: this limit is about simultaneous listeners, not how many books you upload.

07

The system prompt — the agent's standing brief

The system prompt is a paragraph of instructions handed to Gemini at the start of every session. It's the single biggest lever over behaviour, and it lives server-side in the voice-session function (editable in Admin → Agents).

Today it tells the narrator to:

  • Read the chapter verbatim — the exact words, no summarizing.
  • Wrap anything that isn't the book's words — its own remarks, questions, answers — in ((double parentheses)). The browser uses those markers to colour book text black and the agent's asides grey.
  • Pause every few paragraphs and check in; answer reader questions grounded in the chapter.
  • Never say the parentheses out loud.
Why this matters for your decisions

A huge range of behaviour is promptable — pronunciation, how chatty it is, when it pauses, whether it obeys the ((...)) rule, its tone. The test is “could a well-written instruction make it do this?” If yes, it belongs in the prompt (your lever to tune), not in code. Code is only for words getting lost in the pipes or before the mic ever hears them.

08

Where books come from

Before a book can be “talked to,” it goes through a four-step pipeline in the Admin panel:

  1. EPUB — upload the ebook file; the app extracts the clean reading text of each chapter (drops images, page numbers, etc.).
  2. Generate — Gemini writes the chapter list, one-liners, and section titles.
  3. Split — each chapter's text is chopped into ~350-word sections. A check (“coverage”) makes sure 100% of the words survive — an older splitter used to silently drop short paragraphs.
  4. Sync KB — (Cartesia only) pushes sections to Cartesia's knowledge base. Gemini Live doesn't need this step.

All of this ends up inside one big chapters field on the book (a JSONB blob holding the full text + nested sections). Section numbers restart in every chapter, so a section number means nothing without its chapter.

09

Data model & the three stores

The main database tables

  • books — title, author, cover, ratings… and the heavy chapters text blob.
  • profiles — users (who's an admin, etc.).
  • sessions / session_attendees / session_requests — the human narrator bookings (a separate feature from the AI voice).
  • platform_settings — every knob: API keys, chosen provider, model, voice, the prompt. Secrets live here, read only by the server.
  • voice_progress — how far a listener got. voice_events — a debug log of session drops/reconnects.

The three “stores” (the browser's short-term memory)

A store (Zustand) is just the app's in-memory scratchpad while you use it. There are three:

bookStore

public browsing

The catalogue. Now loads only light book info up front; a book's heavy chapter text loads only when you open that book (so the grid stays fast).

adminDataStore

admin panel

All books (incl. unpublished), users, settings. Deliberately skips the heavy chapter text until needed.

adminStore

live UI state

The speech-bubble transcripts, operation progress, pending edits — the things the screen needs moment-to-moment.

10

The other two voice providers

There's a single global switch (voice_provider in Admin → Agents) that picks which voice engine the Talk button uses. Three exist for historical reasons — the product moved and the old ones were never retired:

gemini_live

primary — use this

One live connection does everything. Cheapest, lowest latency, multilingual. The whole guide is about this path.

pipeline

fallback

Old approach: three separate steps (hear → think → speak) chained. Higher latency; kept as a backup.

cartesia

alternate vendor

Different company's agent. Ran out of credits; its modal is the one with the “Mute” button. Slated to be standardized/retired.

Gotcha you already hit

If the switch isn't on Gemini Live, tapping Talk opens a different modal entirely (different layout, a Mute button) and none of the Gemini work applies. Check Admin → Agents first.

11

Secrets, cost & shipping

Where secrets live

API keys sit in platform_settings and are meant to be read only by edge functions on the server.

Known risk (tracked, not fixed)

Today the Gemini key is handed to the browser and put in the connection URL — so a tech-savvy visitor could copy it from their browser tools and run up your Google bill. It only bites at scale, and the fix (proxying the connection through the server) pairs naturally with the Vertex/concurrency work. Flagged, deprioritized.

Roughly what it costs

A full verbatim chapter (~30 min of audio) costs about $0.90 — the spoken audio dominates. India's willingness-to-pay sits near or below that, which is why pricing leans on a cheaper “concise” mode plus per-chapter credits.

How changes go live

  • Frontend (the browser app): push to main or run vercel --prod.
  • Edge function: npx supabase functions deploy <name>.
  • Database / settings: Supabase SQL editor or the Admin panel.

13

Glossary of every abbreviation

Every shorthand used in this guide and in the codebase, in one place.

P2P Peer-to-Peer
People sharing directly with people — here, readers and human narrators, plus the AI.
AI Artificial Intelligence
Software that mimics human tasks — here, understanding and speaking.
LLM Large Language Model
The kind of AI that reads and writes language (Gemini is one).
STT Speech-to-Text
Turning spoken audio into written words (“hearing”).
TTS Text-to-Speech
Turning written words into spoken audio (“speaking”).
Full-duplex
Both sides can talk at the same time — like a phone, unlike a walkie-talkie.
VAD Voice-Activity Detection
Deciding when someone is actually speaking (vs silence/noise).
PCM Pulse-Code Modulation
Raw, uncompressed audio — the format the mic/speaker stream uses.
kHz kilohertz
Audio sampling rate. 16 kHz = 16,000 samples/sec (mic); 24 kHz (playback).
Hz / F0 Hertz / fundamental frequency
Pitch. A weak identity clue on its own — not a fingerprint.
Voiceprint / embedding
A rich numeric “fingerprint” of a voice's timbre — what actually identifies a speaker.
WebSocket (WS)
A connection that stays open both ways — the live “phone line” to Gemini.
API Application Programming Interface
The agreed way one program asks another for something.
SDK Software Development Kit
A ready-made toolkit for using a service in code.
RAG Retrieval-Augmented Generation
Letting the AI fetch relevant text on demand (e.g. other chapters) instead of pre-loading it. Parked.
Context window
How much text the model can hold in mind at once.
Verbatim
Word-for-word, exactly as written — the reading mode we chose.
Developer API vs Vertex AI
Google's simple key-based access (~3 at once) vs its enterprise platform (~1,000 at once).
Concurrency
How many live sessions run at the same time.
JSONB binary JSON
A database column that stores structured data — how chapters is kept.
RLS Row-Level Security
Database rules for who can see which rows.
OAuth Open Authorization
“Log in with Google/LinkedIn” without sharing a password.
OTP One-Time Password
A short code sent to verify you (phone/WhatsApp login — currently hidden).
JWT JSON Web Token
A signed token that proves who a logged-in user is.
FK Foreign Key
A link from one table to another (e.g. a session → its book).
EPUB Electronic Publication
The standard ebook file format we import from.
KB
Knowledge Base (Cartesia's searchable store) — or kilobyte (data size). Context tells which.
MB / KB (size)
Megabyte / kilobyte — a MB is ~1,000 KB. The old catalogue load was multi-MB.
ms milliseconds
Thousandths of a second — the unit for latency and audio padding.
SNR Signal-to-Noise Ratio
How loud the wanted voice is vs background noise. Higher = clearer.
AGC Auto Gain Control
Auto volume levelling — can wrongly amplify background noise in quiet gaps.
WASM WebAssembly
A way to run fast, near-native code in the browser (e.g. a noise remover).
AudioWorklet
A modern, off-main-thread way to process audio in the browser (a planned upgrade).
DOM Document Object Model
The live structure of the page the browser draws.
HMR Hot Module Replacement
Dev feature that live-updates code without a full reload.
UI / UX User Interface / Experience
What the user sees / how it feels to use.
ADR Architecture Decision Record
A logged “why we chose this” entry (see DECISIONS.md).
WTP Willingness To Pay
What a customer will actually pay — central to the pricing story.
Zustand / store
The small library holding the app's in-memory state while you use it.
Edge function
A small program that runs on the server (not the browser) — holds secrets, builds the prompt.
Session resumption / handle
A ticket from Google to reopen and continue a dropped conversation.
Asides · ((double parens))
The markers the narrator wraps its own words in, so the screen can grey them out.
